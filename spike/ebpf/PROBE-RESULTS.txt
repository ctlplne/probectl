S19a eBPF feasibility spike — environment probe
================================================
Captured with bpftool + the kernel config on the probectl build/CI sandbox.
Backs §3 of docs/ebpf-feasibility.md. Raw commands and results:

$ uname -srm
Linux 6.8.0-106-generic aarch64

$ ls -l /sys/kernel/btf/vmlinux
-r--r--r-- 1 nobody nogroup 6960619 /sys/kernel/btf/vmlinux        # kernel BTF present

# Kernel config — every feature CO-RE / kprobe / uprobe needs is compiled in:
CONFIG_BPF=y
CONFIG_BPF_EVENTS=y
CONFIG_BPF_JIT=y
CONFIG_BPF_SYSCALL=y
CONFIG_DEBUG_INFO_BTF=y
CONFIG_FTRACE=y
CONFIG_KPROBES=y
CONFIG_UPROBES=y

$ bpftool btf dump file /sys/kernel/btf/vmlinux format c | wc -l
183000                          # vmlinux.h reconstructs from BTF -> CO-RE viable

$ bpftool btf dump file /sys/kernel/btf/vmlinux format c \
    | grep -cE '^struct (sock|sock_common|inet_sock|tcp_sock|task_struct) '
5                               # the CO-RE structs we need are all in BTF
    struct task_struct {
    struct sock_common {
    struct tcp_sock {

Why the proof can't LOAD here (build toolchain + privilege, not the kernel):
$ clang --version          -> command not found      # cannot compile bpf/*.bpf.c
$ go version               -> command not found      # cannot build the loader
$ id                       -> uid=1002(...) groups=1002(...)
$ capsh --print            -> Current: =             # NO cap_bpf / cap_perfmon / cap_sys_admin
$ ls /sys/kernel/debug/tracing  -> Permission denied

Conclusion
----------
The kernel is fully eBPF/CO-RE capable (BTF present; BPF/KPROBES/UPROBES all =y).
What is missing is the BUILD toolchain (clang) and RUNTIME privilege (CAP_BPF).
That is exactly the "no-kernel CI / unprivileged container / macOS dev" case the
S20 agent must design for — see docs/ebpf-feasibility.md §3 and §11.
